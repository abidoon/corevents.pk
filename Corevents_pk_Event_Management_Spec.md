# Corevents.pk — Event Management Web Application Spec

## 1) Product Overview

**Corevents.pk** is a web application for managing **large-scale corporate events, conferences, trainings, workshops, seminars, and multi-day programs**. It serves two audiences:

1. **Event organizers** — who need to plan, publish, register, communicate, check in, and report on events.
2. **Participants** — who receive access to the application after registering for an event, and use it to view agendas, speaker details, venue information, announcements, certificates, and event resources.

The website hosted on **corevents.pk** should primarily do two things:

- **Explain and showcase the platform** to potential customers.
- **Act as the marketing and trust layer** for the application, with clear calls to action for demos, inquiries, and event onboarding.

The product should feel premium, reliable, modern, and built specifically for **corporate-scale event operations**.

---

## 2) Product Vision

Corevents should become the **single operating system for corporate events**.

It should reduce the operational chaos of:
- registration spreadsheets,
- manual attendee coordination,
- WhatsApp group confusion,
- printed agendas,
- last-minute announcements,
- long queues at check-in,
- and poor post-event reporting.

The platform should centralize the entire event lifecycle:

- event discovery and information,
- registrations and approvals,
- attendee communication,
- agenda and session management,
- speaker and sponsor management,
- badge and QR check-in,
- certificates and feedback,
- analytics and reporting.

## 18) Technical Direction

This section is useful for vibe coding and should guide implementation.

### Suggested stack
- **Frontend:** Next.js
- **Styling:** Tailwind CSS
- **Component library:** shadcn/ui or equivalent
- **Backend:** Laravel, Node.js, or any scalable API stack
- **Database:** PostgreSQL or MySQL
- **Auth:** Email/password + magic link or OTP
- **File storage:** S3-compatible storage
- **Emails:** SMTP or transactional email service
- **QR codes:** generated server-side or client-side
- **PDF generation:** for certificates and reports

### Recommended architecture
- Public website and app can share the same codebase initially.
- Separate marketing pages from authenticated portal routes.
- Use role-based access control.
- Make events multi-tenant if the platform will serve multiple organizations.
